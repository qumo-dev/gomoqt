## 2025-02-25 - [DoS] Out of Memory and Process Crash via Panic in QUIC Varint Decoding
**Vulnerability:** The MOQT message reader used `panic()` when extracting array and slice bounds derived from QUIC varints that hit `math.MaxInt`. In addition, reading `ReadMessageLength` from a raw varint did not validate an upper bounds check against reasonable limits before returning the message size that drives allocation. This meant a malicious client could send extremely large frame lengths or lengths mapping to large arrays, crashing the server process instantly.
**Learning:** Parsing untrusted payload inputs from a network, especially network frame length metadata, should never use `panic()` for bounds validation, and dynamically allocated slices based on those lengths must enforce absolute constraints regardless of CPU architecture constraints (like `math.MaxInt` vs total available RAM).
**Prevention:** Never use `panic()` to handle invalid data bounds in protocol parsers. Instead, return standard errors (e.g., `errors.New("too large")`) so the caller can discard the malformed frame securely without tearing down the process. Ensure absolute upper-bound safety checks on user-provided size allocations (e.g. `size > 50MB`) to mitigate targeted OOM memory exhaustion attacks.
