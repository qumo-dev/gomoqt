## 2024-05-24 - OOM DoS in varint length parsing
**Vulnerability:** Denial of Service (DoS) vulnerability via Out-Of-Memory (OOM) crash. The QUIC varint length prefix for messages is parsed and directly used to allocate a byte slice (`make([]byte, size)`). An attacker could send a message with a massively large length prefix (up to max `uint62`), causing the server to attempt a huge allocation and crash with OOM.
**Learning:** Network length prefixes should never be implicitly trusted for allocating contiguous buffers. We must enforce maximum bounds before allocating. Applying size checks in the payload `Decode` methods immediately prior to `make([]byte, size)` allocation, rather than in the underlying varint reader (e.g., `ReadMessageLength`), preserves the ability for tests and benchmarks to validate maximum integer parsing (e.g., `maxUint62`) without allocating bodies.
**Prevention:** Always enforce an absolute upper boundary (e.g., 50MB limit) for byte slices mapped from network lengths. Return an explicit error like `fmt.Errorf("message too large")`.
