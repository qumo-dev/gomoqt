## 2024-06-14 - Prevent OOM during Varint Decoding

**Vulnerability:** The codebase was vulnerable to DoS/OOM attacks due to unbound data length tracking from untrusted QUIC stream inputs in `ReadMessageLength`, `ReadBytes`, and `ReadStringArray` methods. Specifically, malicious varints could request slice allocations larger than available memory, and `panic` was improperly used to handle large inputs.
**Learning:** `panic()` should never be used as control flow or validation limits when processing data originating from network sources. Network boundary endpoints must enforce strict input size maximums and return standard `error` types that allow the server to gracefully drop connections rather than crash the whole binary. Go's array allocations (`make([]string, 0, count)`) can instantly trigger gigabyte memory requests if bounds aren't validated against buffer size.
**Prevention:** Bound input length tracking against predefined limits (like a conservative 50MB `MaxMessageSize` upper limit) immediately upon QUIC stream reading prior to slice allocations. Ensure `make([]type, count)` where `count` correlates to user input is validated directly against actual remaining bytes available to parse. Return standard errors (`io.EOF`, `errors.New`) instead of using `panic()`.
