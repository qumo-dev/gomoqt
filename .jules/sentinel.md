## 2024-06-19 - Prevent OOM DoS in Network Parsing
**Vulnerability:** The application reads message lengths (varints up to `maxUint62`) directly into slice allocations (e.g., `make([]byte, size)`) without validation. A malicious actor could advertise a massively large message size (e.g., hundreds of MBs or Exabytes) via an 8-byte varint, causing the application to panic or crash via Out-of-Memory (OOM) before any payload was actually transmitted.
**Learning:** This is a classic DoS (Denial of Service) vulnerability in network parsing routines where an untrusted length prefix directly dictates allocation size. Since QUIC's varints max out at `1<<62-1`, this could cause multi-exabyte allocation attempts.
**Prevention:** When parsing network data lengths from varints directly into slice allocations, always enforce a sensible, absolute upper boundary (e.g., 50MB) to prevent exploitable Denial of Service (DoS) conditions via Out-Of-Memory (OOM) crashes from maliciously crafted payloads. If a peer legitimately needs to send larger messages, streaming or chunking should be employed instead of massive contiguous buffers.
