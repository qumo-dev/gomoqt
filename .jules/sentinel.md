## 2024-05-27 - [CRITICAL] Denial of Service via Array Allocation Panic
**Vulnerability:** Found a DoS vulnerability where reading a specially crafted message with an artificially large array `count` size causes a panic and runtime OOM crash in `moqt/internal/message/message_reader.go`'s `ReadStringArray` and `ReadBytes` methods due to unsafe `makeslice` capacities and explicit `panic()` calls on size constraints.
**Learning:** Using untrusted data lengths decoded from network streams directly in `make([]T, 0, untrusted_len)` allocations can cause panics if the size exceeds memory bounds. The condition `count > math.MaxInt` doesn't protect 64-bit systems adequately since max varint value is smaller than `math.MaxInt` yet large enough to trigger an OOM.
**Prevention:** Always validate size/count variables against the remaining buffer length (`uint64(len(remaining_buffer))`) before using them to allocate slices. Never use `panic()` to handle invalid data bounds constraints; gracefully return standard errors (e.g. `io.ErrUnexpectedEOF` or `io.EOF`) to safely handle failure states and protect against Denial of Service attacks.
