## 2025-06-17 - DoS risk via unconstrained length prefix in varints
**Vulnerability:** The system trusted the length prefix read from varints (in `moqt/frame.go` and `moqt/internal/message/message_reader.go`) without reasonable upper bounds, allowing a malicious actor to supply a varint with a huge length which triggers large memory allocations (e.g., `make([]byte, num)`) leading to Out-Of-Memory (OOM) crashes and DoS. Panic was used to handle byte slice length > math.MaxInt, which is also a DoS vulnerability.
**Learning:** `num > math.MaxInt` used `panic()` on bad untrusted inputs. For variables describing slice capacity or size sourced directly from network varints, we must cap the capacity at a sane maximum regardless of the underlying integer capacity limits.
**Prevention:** Never allocate memory blindly based on untrusted length fields. Validate length against logical limits (e.g., 50MB max for typical protocol elements) and available buffer size. Avoid using `panic()` for input validation; instead, gracefully return standard errors like `io.EOF` or an explicit error type.
