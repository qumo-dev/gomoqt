## 2025-02-24 - [CRITICAL] Prevent DoS via Out-of-Memory (OOM) and Panic
**Vulnerability:** The application was vulnerable to Denial of Service (DoS) attacks due to unbounded slice allocations in the varint parser (`ReadMessageLength`) and improper handling of integer overflow limits, which resulted in `panic("byte slice too large")` during message processing.
**Learning:** Network protocol parsing libraries (like MOQT) that parse varints to dynamically allocate payload byte slices must have defensive upper bounds (e.g., a 50MB hard limit) enforced. Untrusted inputs that trigger limits should fail gracefully by returning standard errors instead of panicking, as panics can crash the whole process and be trivially triggered by malicious network streams.
**Prevention:** Always enforce absolute bounds on payload length definitions before memory allocation in custom protocol parsers. Replace `panic()` handlers with returned `error` values for all parsing pathways to avoid total process failures.
